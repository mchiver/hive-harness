# HiveJS Project Review - April 15, 2026


## Summary

HiveJS is a plugin-based Node.js runtime for AI agent harnesses. It provides per-folder data isolation, user authentication, and a tool execution framework with JSON Schema definitions. The project includes three channel implementations (CLI, Web, Discord), 14 plugins, a comprehensive helper library, and 438 passing tests across 35 suites.

This review covers the codebase as of April 15, 2026, evaluating architecture, code quality, testing, security, and areas for improvement.


---


## What Works Well


### 1. Clean Plugin Architecture

The plugin system is the strongest part of the project. The factory pattern (`Plugin.factory.js`) gives each plugin a clear initialization contract, and the automatic entity CRUD generation in `Registry.LoadPlugins()` eliminates boilerplate. Plugins declare an `EntitySchema` once and get `ListEntities`, `ConfigEntity`, `DeleteEntity`, and `RenameEntity` tools for free. This is a genuinely productive abstraction.

The plugin link system (`plugin.link.json`) for external plugin sources is a simple, effective mechanism for development and distribution.


### 2. Consistent Tool Contract

Every tool follows the same shape: `{ ToolName, Description, Parameters, Returns, Execute }`. Parameters use JSON Schema, which makes them self-documenting and directly consumable by LLM tool-use interfaces. The `CommandProcessor` bridges the gap between human CLI input (positional args, `--Key Value`, `Key=Value`, JSON objects) and the structured tool interface. This is well-thought-out and practical.


### 3. Channel Abstraction

The `Channel` base class (`Source/Channel.js`) cleanly separates shared behavior (argument parsing, conversation resolution, input routing, autocomplete, dry-run) from channel-specific I/O (CLI readline, Express HTTP, Discord bot). Adding a new channel requires implementing ~6 abstract hooks. The input routing logic (slash commands, tool invocations, chat fallback) is clear and well-ordered.


### 4. EventBus Integration

The `EventBus` with wildcard pattern support (`tool.*`, `conversation.**`) is woven through the tool invocation pipeline and conversation system. This enables SSE streaming in the web channel and audit hooks without coupling components. The sequential await ordering ensures predictable handler execution.


### 5. Comprehensive Test Coverage

438 tests across 35 suites covering helpers, core system, all plugins, all channels, and the Exchange simulation. Tests use Node's built-in test runner (no framework dependency). The numbered file naming (`050-Humanize.test.js`, `100-Registry.test.js`, `200-System-Tools.test.js`) enforces execution order and makes the test progression easy to follow. All tests pass cleanly.


### 6. Conversation Pipeline

The `Conversation.Chat` tool implements a complete LLM interaction loop: topic-based context retrieval, chat history, skill injection, prompt construction, tool call parsing, multi-step tool execution with continuation prompts, and full history recording. The prompt is built as structured XML with `<conversation>`, `<skill>`, `<context>`, `<history>`, and `<task>` sections. This is a solid foundation for AI agent integration.


### 7. Exchange Plugin

The virtual financial exchange is impressively complete: limit order book matching with price-time priority, maker-price execution, account/holding management, asset registration, participant roles (supplier, consumer, speculator, hybrid), manufacturing, liquidation ticks, and a dynamic skill that teaches LLMs how to trade. The matching engine in `Exchange.factory.js:MatchOrders()` handles partial fills correctly.


### 8. Helpers Library

The helpers are small, focused, and dependency-free where possible:
- `FileUtils` centralizes all filesystem operations behind a single interface
- `Humanize` handles duration parsing/formatting (`"1h30m"` <-> milliseconds)
- `Strings` provides glob-to-regex and glob-to-SQL-LIKE conversion
- `SqlStore` wraps better-sqlite3 with a clean open/query/execute/schema API
- `Fetch` uses only Node built-ins (no axios/node-fetch dependency)
- `Logger` provides severity levels, colorization, and message truncation


### 9. Minimal Dependencies

The project has only 7 runtime dependencies: `bcrypt`, `better-sqlite3`, `blessed`, `discord.js`, `express`, `jsonwebtoken`, `minimist`, and `moniker`. No TypeScript, no build step, no bundler. This keeps the project simple and reduces supply chain risk.


### 10. Coding Style Consistency

The codebase follows its own style guide consistently: tab indentation, PascalCase for parameters and external functions, `lower_case` for locals, spaces inside brackets, braces on new lines, and visual separator comments. The code reads cleanly despite being plain JavaScript.

## What Could Be Improved


### 1. Security: `parse_relaxed_json` Uses `new Function()`

**File:** `Helpers/CommandProcessor.js:318`

```javascript
var fn = new Function( 'return (' + Text + ')' );
return fn();
```

This evaluates arbitrary JavaScript to parse "relaxed JSON" (unquoted keys, etc.). In the CLI channel this is a convenience feature, but in the Web channel, user-supplied arguments reach `CommandProcessor.Parse()` through the tool invocation route. An attacker with a valid JWT could craft a tool call with a malicious argument string that executes arbitrary code on the server. Consider using a safe JSON5 parser or restricting relaxed parsing to CLI-only contexts.


### 2. Security: Web Channel Shared Hive Instance

**File:** `Channels/Web/Web.js:66-74`

The web channel stamps the requesting user's identity onto a shared `Hive` object per-request. The comment acknowledges this relies on Node's single-threaded model, but `async/await` does yield to the event loop. If two requests arrive close together, the second request's middleware could overwrite `Hive.UserName` before the first request's tool execution reads it. This is a race condition that could cause one user to execute tools under another user's identity.

**Recommendation:** Create a per-request `Hive` proxy or clone that carries its own user context, or pass user identity through the tool invocation chain rather than mutating shared state.


### 3. Security: JWT Signed with Password Hash

**File:** `Source/Registry.js:265`

Using the user's bcrypt password hash as the JWT signing secret means that all tokens for a user share the same secret, and if the user file (which stores the hash) is readable, tokens can be forged. A separate, randomly-generated server secret stored in the registry config would be more standard.


### 4. Security: No Role-Based Access Control Enforcement

Plugins declare `RequiredRole` (e.g., `'user'`, `'guest'`), but the `CommandProcessor.Invoke()` method never checks the user's role against the plugin's required role. Any authenticated user can invoke any tool regardless of their role assignment. The role field is present but not enforced.


### 5. No Input Validation on Entity/Plugin Names

Entity names and plugin names flow directly into filesystem paths (`PATH.join( Hive.DataPath, PluginName, EntityName )`). Names containing `..`, `/`, or `\` could traverse outside the intended data directory. The Web channel's entity routes pass `req.params.name` directly to `ConfigEntity` without sanitization.

**Recommendation:** Validate that entity and plugin names contain only safe characters (alphanumeric, hyphens, underscores) before using them in path construction.


### 6. Database Connection Management

Plugins that use SQLite (Conversation, Exchange, MessageQueue, Workflow) open and close database connections per tool call via `OpenDatabase()` / `OpenHistoryDatabase()`. Each call creates a new `better-sqlite3` connection, runs the operation, and closes it. For high-frequency operations (e.g., chat messages), this adds overhead and prevents connection reuse.

**Recommendation:** Consider connection pooling or keeping a single connection per entity open for the lifetime of the Hive instance, closing on shutdown.


### 7. Duplicated `GetEntityConfig` Pattern

Every entity-type plugin (Conversation, Llm, Exchange, Workflow) implements its own `GetEntityConfig` method with nearly identical code: resolve the entity folder, build the config path, check if the file exists, read and return JSON. This could be a shared method on the `Plugin` base class or the `Entities` helper, reducing duplication across 4+ plugins.


### 8. Error Handling Inconsistency

Some tools return `{ Error: "message" }` inside a successful result envelope, while others throw errors that get caught by `CommandProcessor.Invoke()` and wrapped in `{ Success: false, Error: "message" }`. The Chat tool, for instance, returns `{ Error: "..." }` for LLM failures (which appears as `{ Success: true, Result: { Error: "..." } }`). Callers must check both `result.Success` and `result.Result.Error`, which is fragile. A consistent error contract would reduce bugs.


### 9. `EnsureTables` Called on Every Database Open

Both `Exchange.OpenDatabase()` and `MessageQueue.OpenDatabase()` call `EnsureTables()` every time they open a connection. Since table creation uses `CREATE TABLE IF NOT EXISTS`, this is safe but wasteful. The Conversation plugin has the same pattern but checks `store.ListTables()` first, adding a query per open. Consider tracking initialization state to skip this check on subsequent opens.


### 10. Workspace Plugin Path Handling

The Workspace plugin tools operate on the Hive root directory, but `ResolvePath` is referenced in `Sed.js` without being shown in the factory. If `ResolvePath` simply joins the hive root with the user-supplied path, it's another path traversal vector. The `ReadFile`, `WriteFile`, `DeleteFile`, etc. tools should validate that the resolved path stays within the hive root.


### 11. Discord Channel Awaiting in Fire-and-Forget

**File:** `Channels/Discord/Discord.js:226`

The `Output()` method calls `discord_channel.send()` in a synchronous loop without awaiting. Since `send()` returns a Promise, messages could arrive out of order for multi-chunk replies. Either await each send or collect the promises and await them together.


### 12. Missing `package.json` Description and Author

The `package.json` has empty `description` and `author` fields. If this is intended for distribution (via npm or otherwise), these should be filled in.


### 13. Web Channel Plugin Config Route Writes Directly

**File:** `Channels/Web/Routes/Entities.js:118`

The `PUT /api/plugins/:plugin/config` route writes `req.body` directly to the plugin config file and merges it into the live plugin object with `Object.assign`. There is no schema validation, so a client could inject arbitrary properties into the plugin object, potentially overwriting methods or internal state.


### 14. No Graceful Shutdown

The CLI channel calls `process.exit(0)` in `Stop()`, and the Discord channel does the same. There's no cleanup of open SQLite connections, pending EventBus handlers, or in-flight tool executions. For the web channel, `Server.close()` is called, but open SSE connections and database handles are not explicitly cleaned up.

## Architectural Observations


### Strengths of the Current Design

- **Flat plugin loading** keeps things simple. No dependency resolution, no lifecycle hooks beyond `Initialize()`. Plugins are independent units.
- **JSON Schema everywhere** creates a unified contract between plugins, tools, the CLI, the web UI, and LLM consumers.
- **SQLite per entity** provides good isolation. One entity's data corruption doesn't affect others.
- **No build step** means the codebase is what runs. No compilation surprises, no source map debugging.


### Areas to Watch as the Project Grows

- **Plugin-to-plugin dependencies** are currently handled by tools calling other tools via `Hive.InvokeTool()`. This works but creates implicit coupling. If plugin load order ever matters, the current flat loading won't handle it.
- **The Conversation plugin** is becoming a large surface area (prompt building, history management, skill resolution, topic search, tool call parsing). Consider whether some of these responsibilities should be separate components.
- **Web frontend** uses AngularJS + Bootstrap with a traditional MPA structure. The SPA fallback and SSE streaming are solid, but the frontend JavaScript lacks the same test coverage as the backend.

## Recommendations (Priority Order)

1. **Fix the `new Function()` eval** in CommandProcessor - this is the most exploitable issue.
2. **Resolve the shared Hive user context race** in the web channel.
3. **Add path traversal protection** for entity names and workspace paths.
4. **Enforce `RequiredRole`** checks in tool invocation.
5. **Standardize error returns** (always throw or always return `{ Error }`, not both).
6. **Extract `GetEntityConfig`** into shared infrastructure to reduce plugin duplication.
7. **Add connection management** for frequently-opened SQLite databases.
